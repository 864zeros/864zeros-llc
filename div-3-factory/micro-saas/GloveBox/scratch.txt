**The cutting-edge pattern for 2026: API responses ARE KG fragments.**

You don't aggregate into a flat object. You aggregate into a graph. Each API response becomes typed nodes. The aggregation IS the DAG process. The service doesn't return JSON — it returns a capsule.

```
NHTSA VIN decode  →  nodes: make, model, engine, safety_features
NHTSA recalls     →  nodes: recall_id, component, severity, remedy
EPA fuel data     →  nodes: mpg_city, mpg_highway, emissions
Carfax PDF parse  →  nodes: service_events, accidents, ownership_chain
User receipts     →  nodes: oil_change_date, tire_rotation, brake_service

        ↓ aggregate ↓

   Single typed JSON-LD graph = the car's KG
   Narrative synthesis = the car's KB
   First-person voice = the car's persona
   
        ↓ stamp ↓
   
   GLOVEBOX capsule. The car is an agent.
```

**The architecture:**

| Layer | What | Tech |
|-------|------|------|
| API Gateway | Single entry point for all 864z apps | SAP CAP / Node.js |
| Adapters | One per external API, normalizes response | Typed transform functions |
| Composer | Merges adapter outputs into JSON-LD graph | **This is DAG's K function** |
| Store | Persists the composed graph | SQLite / HANA / filesystem |
| Service | Exposes composed object via OData/REST | CAP service layer |
| Consumer | AETHER capsule OR any other 864z app | Capsule stamper / UI |

**Where Carfax gets its data:**

| Carfax Source | Can We Access? | How |
|-------------|---------------|-----|
| NHTSA (recalls, complaints, safety) | **YES — FREE** | Public API, no key |
| NMVTIS (National Motor Vehicle Title Information System) | **YES — $2-5/query** | Approved consumer providers (VinAudit, ClearVin) |
| State DMV records (title, registration, odometer) | **Partial** — varies by state | Some states have APIs, most don't |
| Insurance claims (accidents, total loss) | **NO** — private data sharing agreements | Carfax has exclusive contracts with insurers |
| Dealership service records | **NO** — Carfax pays dealers to report | Proprietary network |
| Independent shop service records | **Partial** — some report to Carfax, some to OpenBay/RepairPal | Fragmented |
| Auction records (Manheim, ADESA) | **NO** — wholesale only | Dealer access only |
| Manufacturer warranty/recall repairs | **YES via NHTSA** for recalls, **NO** for warranty | Warranty data is OEM-proprietary |
| State inspection stations | **Partial** — some states publish | State-specific |
| Fleet/rental records | **NO** | Enterprise data sharing |
| **OBD-II (the car itself)** | **YES — FREE** | $15 Bluetooth dongle + app |

**The breakthrough: OBD-II.**

Your car has a computer. It knows everything. Diagnostic trouble codes, engine hours, coolant temp, battery voltage, trip data, pending fault codes. A $15 ELM327 Bluetooth dongle + a Python library (`python-obd`) reads it all. This is data Carfax will NEVER have because they don't talk to the car directly.

**What we can aggregate WITHOUT Carfax:**

| Source | Data | Cost |
|--------|------|------|
| NHTSA VIN decode | Specs, safety equipment | Free |
| NHTSA Recalls | Open recalls for this VIN | Free |
| NHTSA Complaints | Known problems for this model/year | Free |
| EPA | Fuel economy, emissions | Free |
| NMVTIS provider | Title history, brands, odometer readings, total loss | $2-5 one-time |
| OBD-II dongle | Live diagnostics, DTCs, engine data, mileage | Free after $15 dongle |
| Manufacturer maintenance schedule | Service intervals by mileage | Scrapeable / static data |
| User-uploaded Carfax PDF | Baseline service history | User's existing purchase |
| User-uploaded receipts | Ongoing service tracking | Free |

**That covers 80% of what Carfax sells.** The 20% we miss (insurance claims, dealer service networks, auction history) is their moat. But for an OWNER who has the car, OBD-II gives us data Carfax doesn't even have.
